# Производительность

`@ambrosia/events` спроектирован для максимальной производительности. Вот ключевые принципы и рекомендации.

## Архитектура "build once, emit fast"

Вся тяжёлая работа выполняется один раз при старте приложения:

| Операция | Когда | Стоимость |
|----------|-------|-----------|
| Сканирование `@OnEvent` метаданных | `onInit` | O(n) провайдеров |
| `.bind()` методов к инстансам | `onInit` | O(n) обработчиков |
| Сортировка по приоритету | `onInit` | O(n log n) на событие |
| `emit()` — lookup | Runtime | O(1) — `Map.get()` |
| `emit()` — вызов обработчиков | Runtime | O(k) — прямой for-цикл |

## Sync vs Async

EventBus автоматически определяет, какие события имеют **только синхронные** обработчики, и использует быстрый путь:

### Синхронный fast path

Если **все** обработчики события синхронные, `emit()` возвращает `void` (не `Promise`):

```ts
@Injectable()
class MetricsListener {
  @OnEvent(RequestEvent)
  onRequest(event: RequestEvent) {
    // Синхронный обработчик — без async overhead
    this.counter++;
  }
}

// emit() возвращает void — ноль Promise аллокаций
bus.emit(new RequestEvent()); // sync!
```

### Асинхронный путь

Если хотя бы один обработчик асинхронный, используется `Promise.all`:

```ts
@Injectable()
class EmailListener {
  @OnEvent(UserRegisteredEvent)
  async sendEmail(event: UserRegisteredEvent) {
    // Async — триггерит Promise path
    await this.mailer.send(event.email, 'Welcome');
  }
}
```

### Рекомендация

Если событие не требует async-обработки — используйте синхронные обработчики для максимальной скорости.

## Приоритеты

Обработчики сортируются по приоритету один раз при инициализации, а не при каждом `emit()`:

```ts
@OnEvent(OrderEvent, { priority: 100 })  // Выполнится первым
validateOrder(e: OrderEvent) {}

@OnEvent(OrderEvent, { priority: 50 })   // Вторым
chargePayment(e: OrderEvent) {}

@OnEvent(OrderEvent)                      // Последним (priority: 0)
sendConfirmation(e: OrderEvent) {}
```

Используйте приоритеты для гарантии порядка выполнения (валидация перед отправкой).

## Fire and Forget

Для событий, результат обработки которых не важен, используйте `emitAsync()`:

```ts
// НЕ ждёт завершения обработчиков
this.bus.emitAsync(new AnalyticsEvent('page_view', { url: '/home' }));
```

Это полезно для аналитики, логирования и других side-effects, которые не влияют на основной flow.

## Советы по оптимизации

1. **Избегайте тяжёлых операций в sync-обработчиках** — они блокируют emit
2. **Группируйте связанные обработчики в один метод** — меньше вызовов функций
3. **Используйте `emitAsync()` для non-critical events** — не блокирует вызывающий код
4. **Минимизируйте данные в событиях** — передавайте ID вместо полных объектов
5. **Не создавайте обработчики, которые ничего не делают** — каждый обработчик это вызов функции
